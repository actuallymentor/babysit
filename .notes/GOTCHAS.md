# Gotchas

Accumulated from sir-claudius + babysit development.

1. **Two-phase OAuth** — detect creds existence *before* pre-flight auth check. Pre-flight can rotate tokens. Capture *after* pre-flight.
2. **Docker inode tracking** — never `mv` a bind-mounted credential file. Docker tracks the original inode. Use `writeFileSync` (in-place) instead.
3. **ANSI cursor-forward** — `\x1b[nC` is used as visual whitespace by TUI agents. Replace with real space *before* stripping ANSI, or pattern matching breaks.
4. **Tmux session name limit** — ~256 chars max. Hash long paths (>200 chars) using SHA-256 truncated to 16 chars.
5. **Debounce auto-accept** — 3s minimum between consecutive auto-accepts. TUI redraw flicker causes false double-fire.
6. **Sync daemon cleanup** — use `unref()` on detached children, not `disown`. Avoids "Terminated" message on stderr.
7. **node_modules isolation** — Linux container produces Linux-native binaries. Mount a docker volume over `/workspace/node_modules` instead of bind-mount to prevent host breakage.
8. **Claude plan acceptance** — post-v2.1, plan accept is `\x1b[Z` (Shift+Tab), NOT Enter. Enter now rejects.
9. **mri's `unknown` callback halts parsing** — when set, mri returns the callback's return value instead of the parsed result. Don't use it for "include unknown args" — omit `unknown` entirely and identify known flags after parsing.
10. **tmux `send-keys -l` consumes all remaining args** — `Enter` after a `-l text` arg gets sent as literal text, not a key press. Send the literal text and the Enter in two separate `send-keys` calls.
11. **tmux `send-keys` shift+tab** — pass the named key `BTab`, not the literal escape string `\x1b[Z`. tmux interprets unrecognised key names as text input.
12. **Don't bind-mount the host's `~/.claude/settings.json`** — Claude rewrites it at runtime and would overwrite the user's host settings. Build a tmpfile that merges host settings + babysit overrides and mount that instead.
13. **`build_docker_command` returns a string consumed by `sh -c`** — every value-bearing arg (system prompt, env values, paths with spaces) MUST be shell-quoted. We use a single-quote wrap with `'\''` escaping for embedded quotes. Tests in `tests/docker.test.js` guard this; do not regress to a plain `flags.join(' ')`.
14. **Codex `resume` is interactive, `exec resume` is non-interactive** — babysit supervises through tmux, so the codex adapter must use `[ 'resume', id ]`. The `exec resume` form would launch in a non-interactive mode that can't be supervised.
15. **Docker image namespace is `actuallymentor/babysit`, not `babysit/babysit`** — both the publish workflow and the runtime pull/run must agree. Source in `src/docker/update.js` was originally wrong; tests in `tests/docker.test.js` lock this in.
16. **System prompt for non-claude agents goes via env + entrypoint** — codex/gemini/opencode have no `--append-system-prompt` flag, so the docker run injects `BABYSIT_SYSTEM_PROMPT` and `BABYSIT_SYSTEM_PROMPT_FILE` and the entrypoint writes the right file in `/home/node/.codex|.gemini|.config/opencode/`. Use container-local paths (NOT `/workspace/...`) — `/workspace` is read-only in mudbox and ephemeral in sandbox, so the write would silently fail there.
17. **mri normalises `--no-X` to `{X:false}`, not `{'no-X':true}`** — listing `'no-update'` in `boolean:` makes mri quietly drop the flag and store `update:false` instead. Detect with `args.update === false`. Symptom: `--no-update` was silently ignored, self-update always ran.
18. **Bun-compiled binaries can't `readFileSync` source-tree assets** — `__dirname` resolves to `/$bunfs` at runtime, so `readFileSync(join(__dirname, '../foo'))` errors with ENOENT for both `package.json` and our `system_prompt/*.md`. Use import attributes instead: `import x from './x.json' with { type: 'json' }` for JSON, or move text content into a `.js` module and import it normally. (Bun supports `with { type: 'text' }` but Node 24 does not, so the JS-module pattern is the cross-runtime safe choice.)
19. **Codex global instructions live at `${CODEX_HOME}/AGENTS.md`, NOT `instructions.md`** — older codex versions read `~/.codex/instructions.md`, but current docs only honor `AGENTS.override.md` then `AGENTS.md` at the global scope. Writing to `instructions.md` silently does nothing. Babysit was bitten by this between v0.3.0 and v0.3.2 — codex was launching without any of the babysit system prompt.
20. **Gemini's `GEMINI_CLI_HOME` is a parent dir, not the config dir** — gemini creates `.gemini/` *inside* whatever `GEMINI_CLI_HOME` points to. Setting it to `/home/node/.gemini` (the actual config dir) would have gemini look at `/home/node/.gemini/.gemini/` for everything. Set it to the parent (`/home/node`) instead. Codex / claude / opencode use the opposite convention — their env vars point at the config dir directly.
21. **Each agent's credentials live in a different shape** — `keychain_service`-only logic in `credentials/darwin.js` will silently drop adapters that declare just a `file:` field (opencode does this). Always handle keychain → fallback_file → standalone file → env_key as four independent paths, and let env_key stack additively on top of file-based creds.
22. **OAuth file paths per agent** — claude: `~/.claude/.credentials.json` (linux) or Keychain `Claude Code-credentials` (darwin, with file fallback). codex: `~/.codex/auth.json`. gemini: `~/.gemini/oauth_creds.json`. opencode: `~/.local/share/opencode/auth.json` (no Keychain on any platform — opencode doesn't use it).
23. **Pre-flight rotation must run between detect and capture** — sir-claudius lesson; without it a near-expiry token rides the container until the 5-min sync daemon catches up. We invoke `<agent> --version` on the host so the agent's own refresh logic fires before we copy the file/keychain entry.
24. **Pre-create every credential dir in the Dockerfile** — when `docker run -v /tmp/foo:/home/node/.codex/auth.json` lands and `/home/node/.codex` doesn't exist in the image, docker auto-creates the parent as root, which blocks the `node` user from writing refreshed tokens or babysit-generated prompt files into it. Add the dir to the `mkdir -p` list in `Dockerfile` *before* the `chown -R node:node /home/node` line.
25. **Claude `--sandbox` cannot bind-mount writable session dirs** — `~/.claude/{projects,plans,todos}` is where claude writes per-session JSON. Mounting host paths read-only crashes claude on first write; bind-mounting read-write would touch host state in a mode that's supposed to be ephemeral. Skip the mounts entirely in sandbox so claude writes inside the container.
26. **The supervisor must outlive the foreground cli** — the user expects `babysit <agent>` to drop them into the agent's tmux session, then detach with Ctrl+B d and have the supervisor + container keep running. If the monitor loop runs in the foreground process, it dies the moment the user detaches and the agent goes unsupervised (and credentials stop refreshing). Pattern: spawn `babysit __monitor <id>` as a detached child (`{ detached: true, stdio: 'ignore' }` + `unref()`), then `execSync` `tmux attach` in the foreground. The detached daemon re-loads session metadata from `~/.babysit/sessions/<id>.json` to reconstruct config/agent/patterns/cwd.
27. **`config.commands` execSync freezes the supervisor** — running babysit-yaml-defined shell commands via `execSync` blocks Node's event loop, which means pane capture and rule evaluation freeze for as long as the command runs. A user-supplied notify_command that does a slow curl can pause supervision for seconds. Use `spawn` (with `stdio: 'inherit'`) and await its `close` event instead.
28. **bun-compiled binary detection via `process.argv[1]`** — node sets it to the script path (e.g. `src/index.js`); bun-compiled binaries set it to a synthetic `/$bunfs/root/<basename>` path. Check `argv[1]?.startsWith('/$bunfs')` to detect the compiled mode when re-spawning self. `process.execPath` reliably points at the right binary in either case.
29. **Tmpfile mount perms — `writeFileSync(path, content, { mode: 0o666 })` is masked by umask** to 0o664 (umask 002) or 0o644 (umask 022). The container's `node` user (uid 1000) is neither the host file's owner nor in any matching group, so it lands at "other" perms with no write. Some agents (claude, codex, gemini) update their state file in place during init — when that write silently fails, the TUI can hang without ever finishing first render. **Always follow `writeFileSync` with `chmodSync(path, 0o666)`** when the tmpfile is bound into the container; relying on the `mode` option is silently wrong. Symptom: claude renders blank pane and never shows the welcome screen, even though `claude --print` works inside the same container. The shared helper lives in `src/utils/tmpfile.js`; use it instead of re-implementing the write+chmod pair.
30. **Claude `.claude.json` carries onboarding/trust state** — without `hasCompletedOnboarding: true`, a `projects[/workspace].hasTrustDialogAccepted: true` entry, AND a high `lastOnboardingVersion`, claude treats every fresh container as a brand-new install (or as "version-upgraded since onboarding"), pops the theme picker AND the workspace-trust dialog. Neither has a CLI flag override (`--dangerously-skip-permissions` only affects tool approvals, not onboarding/trust). The `lastOnboardingVersion` gate is what regressed in claude ≥ 2.1.x — `hasCompletedOnboarding` alone is no longer enough, because claude reruns the version-delta onboarding (theme picker) whenever its installed version is newer than the recorded `lastOnboardingVersion`. The Dockerfile pulls latest claude on every image build, so the host's recorded version is almost always behind. Babysit copies the host's `~/.claude.json` to a tmpfile, sets all three flags (version pin uses the `ONBOARDING_VERSION_SENTINEL` constant — `9999.0.0` — so future releases stay covered), and bind-mounts it at `${CLAUDE_CONFIG_DIR}/.claude.json` inside the container. See `claude_extra_mounts` in `src/agents/setup.js`.
31. **Claude `~/.claude/{projects,plans,todos}` should be named docker volumes, NOT bind mounts** — bind-mounting hits two failure modes back-to-back. (a) The host dirs typically have 0770 owner perms, which the container's `node` user can't write, so claude renders an EACCES error mid-chat. (b) If the agent works around (a) by `sudo chown -R node:node /home/node/.claude/plans`, the chown propagates back to the host bind mount and silently changes ownership of the user's host dirs — wrecking subsequent host-side claude runs. Named volumes (`babysit-claude-{projects,plans,todos}:/home/node/.claude/...`) sidestep both: claude has full write access, host dirs aren't touched, `babysit resume` still finds the session because the volume persists across container restarts.
32. **Codex trust + model nags persist per-install** — the trust dialog ("Do you trust the contents of this directory?") is bypassed by adding `[projects."/workspace"]\ntrust_level = "trusted"` to the mounted `~/.codex/config.toml`. The "Try new model" intro for each gpt-5.x release is bypassed by adding entries under `[tui.model_availability_nux]` (e.g. `"gpt-5.4" = 2`). Both nags would otherwise stall the supervised session on every fresh container. Babysit's `codex_extra_mounts` injects both. The `CODEX_KNOWN_MODELS_FOR_NUX` list needs maintenance as new gpt versions ship.
33. **Codex `installation_id` mount triggers "Failed to create session: Operation not permitted"** — empirically, mounting `~/.codex/installation_id` from the host into the container makes codex's session-tree machinery fail with EPERM on `/home/node/.codex/sessions`. Cause unclear (likely io_uring + seccomp, or a per-install file-locking scheme), but the fix is to NOT mount installation_id and let codex generate a fresh one per container. There's a regression test in `tests/agents.test.js` that asserts `codex_extra_mounts()` excludes installation_id.
34. **Gemini "logged in" state spans multiple files** — `~/.gemini/oauth_creds.json` alone is not enough. Without `~/.gemini/settings.json` (which carries `auth.selectedType: "oauth-personal"`), gemini drops into its auth-method picker on launch and ignores the OAuth tokens entirely. babysit's `gemini_extra_mounts` mounts `settings.json`, `google_accounts.json`, `installation_id`, and `state.json` as tmpfile copies, plus a synthesized `trustedFolders.json` with `/workspace: TRUST_FOLDER`. The CLI flag `--skip-trust` is also wired (`extra_args`) as a yolo-only belt-and-braces guard.
35. **Don't force `gemini-pro-latest` on free-tier users** — the alias 404s with "Model 'gemini-pro-latest' was not found or is invalid" for users on Code Assist for Individuals (Pro models are restricted to paid plans now). Letting gemini's internal "agent router" pick the model based on the user's plan (`defaults: {}` in the gemini adapter) is more portable than guessing. The router currently routes free-tier users to `gemini-3-flash-preview`.
36. **Opencode default `gpt-5.5-pro` is rejected by ChatGPT-account auth** — error: "Bad Request: The 'gpt-5.5-pro' model is not supported when using Codex with a ChatGPT account." Pinning babysit's opencode `defaults.model` to `openai/gpt-5.5` works for both OAuth and API-key auth paths. Users with non-openai providers (anthropic, google) override via passthrough `--model anthropic/claude-opus-4-7` (etc.).
37. **Codex `apps` feature triggers a noisy `codex_apps` MCP token-expired warning on every startup.** The `apps` feature (default-on, stable per `codex features list`) spawns the codex_apps MCP, which connects to OpenAI's hosted Drive/Slides/etc connectors and demands a fresh OAuth access token at startup. Babysit's pre-flight refresh (`<agent> --version`) does NOT actually refresh the codex token — confirmed empirically: `auth.json` mtime is unchanged after `codex --version`, `codex login status`, `codex mcp list`, and similar non-session commands. So any container started >1h after the user's last interactive codex run reliably hits `MCP client for codex_apps failed to start: ... token_expired`. Connectors are useless inside a sandboxed coding-agent context anyway, so we disable the feature in babysit's mounted `config.toml` via `[features]\napps = false` (equivalent to passing `--disable apps`). See `inject_codex_first_run_bypass` in `src/agents/setup.js`. To re-enable for a specific session: `babysit codex -- -c features.apps=true` (passthrough).
38. **Codex warns and falls back to a vendored bubblewrap when the system one is missing.** The base node:24-slim image doesn't include bubblewrap, so codex prints `⚠ Codex could not find bubblewrap on PATH` on every container startup before falling back. Add `bubblewrap` to the apt-get list in `src/docker/assets/Dockerfile` to silence it. The vendored fallback works fine, but the warning looked alarming enough that users asked about it.
39. **Install/update target is `~/.local/bin/babysit`, not `/usr/local/bin/babysit`.** The original install.sh used `/usr/local/bin` (root-owned, sudo-required), which meant `babysit update` had to escalate via `sudo mv` on every binary swap. Switched both `scripts/install.sh` and `src/cli/update.js` to write to `~/.local/bin` so the entire install/update lifecycle is sudo-free. Migration handling: when `process.execPath` lives in a non-user-writable dir (legacy install), `babysit update` writes the new binary to `~/.local/bin` and prints a one-time notice telling the user to `sudo rm` the legacy copy themselves — we never escalate automatically. PATH ordering matters: `~/.local/bin` must come before `/usr/local/bin` or the legacy binary keeps shadowing the new one. Keep the constant in update.js (`USER_INSTALL_DIR`) in sync with install.sh's `INSTALL_DIR`.
40. **`skipDangerousModePermissionPrompt` is top-level in claude's settings.json, NOT nested under `permissions`.** Claude only honours the field at the root. Putting it under `permissions: { skipDangerousModePermissionPrompt: true }` looks plausible (it's adjacent to `defaultMode`, `allow`, `deny`, etc.) but silently does nothing — the warning dialog still fires. Babysit injects this into the merged settings tmpfile when `mode.yolo` is on; see `build_claude_settings_tmpfile` in `src/agents/setup.js`. The field is also ignored when set in project settings (`.claude/settings.json`); it must be in user-scope settings, which is what babysit's tmpfile mount provides via `/home/node/.claude/settings.json`.
41. **Pre-flight `<agent> --version` does NOT actually rotate codex/gemini/opencode tokens.** Claude refreshes on `--version` (the assumption from gotcha #23 / #34), but codex/gemini/opencode only refresh on real API calls — empirically confirmed: `auth.json` mtime is unchanged after `<agent> --version`. So pre-flight is a silent no-op for those three. The host's auth.json keeps the original refresh_token, babysit copies it, the container refreshes (rotating one-time-use refresh_token X → Y), writes Y to the bind-mounted tmpfile but NOT back to the host file. Next babysit session reads the now-server-invalidated X from host → first refresh attempt fails with `"Your access token could not be refreshed because your refresh token was already used."` Fix: bidirectional credential sync — `start_credential_sync` takes a `write_destination` callback and pushes container-side updates back to the host file on every tick + on `stop()`. Conflict policy is source-wins (host re-auth beats container refresh). Keychain-backed creds (claude on darwin) intentionally stay one-way because claude's pre-flight rotation works reliably and writing back to Keychain has a different shape than a file write. Recovery for users hitting this on an already-invalidated host file: re-auth on the host (`codex auth login` etc.).
42. **Value-flag normalization must feed passthrough collection too.** For `--log`, parse.js rewrites a bare flag to `--log=` before giving argv to mri so `--log --yolo` does not consume `--yolo` as a value. `collect_passthrough` must use that same prepared argv. If it uses the original argv, it will still treat bare `--log` as a value-taking flag and drop the next passthrough token; e.g. `babysit claude --log --model sonnet` becomes just `sonnet` for the agent.
43. **Do not type the docker boot command into tmux when setting up early `pipe-pane` logging.** `send-keys -l "exec ${command}"` would capture startup output, but it also echoes the full docker command into the pane/log, including env values. Start the pane with a shell that waits on `read`, pass the command as a shell argument, attach `pipe-pane`, then send a blank Enter so no command text is echoed before execution.
44. **The monitor daemon must reuse the foreground's credential tmpfile, not mint its own.** `copy_host_file_to_tmpfile` bakes `Date.now()` into the path so two consecutive calls return DIFFERENT paths. The container is started with the foreground's tmpfile bind-mounted; if the monitor calls `setup_credentials(agent)` with no options, it creates a brand-new tmpfile_M and starts a sync on tmpfile_M — but the container is writing OAuth refreshes to tmpfile_FG. tmpfile_FG goes unwatched, host `auth.json` never gets updated, and the next `babysit codex` reads the stale (now server-invalidated) refresh_token → fails immediately with "Your access token could not be refreshed because your refresh token was already used." This silently neutralised the bidirectional sync from gotcha #41 for ~one release. Fix: `setup_credentials` accepts `{ existing_tmpfile }`; `cmd_monitor` passes `session.creds_tmpfile` so its sync watches the same path the container is mounting. Regression test in `tests/credentials_setup.test.js`.
45. **Credential-sync monitor handoff needs the foreground capture baseline.** Reusing the same tmpfile is not enough if codex refreshes immediately on startup before `__monitor` has established its sync. If the monitor seeds both `last_*_hash` values from the already-rotated tmpfile, the still-stale host `~/.codex/auth.json` looks like a fresh host-side reauth and source-wins logic overwrites the valid tmpfile token. Store safe SHA-256 hashes from the foreground capture in session metadata (`creds_sync_baseline`) and pass them to the monitor; then the startup tmpfile rotation is detected as tmpfile → host instead of host → tmpfile.
46. **Codex host credentials follow host `CODEX_HOME`, not always `~/.codex`.** Babysit pins in-container `CODEX_HOME=/home/node/.codex`, but credential capture must resolve the host's live Codex home first. If the user runs host Codex with `CODEX_HOME=/Users/mentor/.agents/`, hardcoding `~/.codex/auth.json` copies stale auth into the container and reproduces the "refresh token was already used" failure even after the sync race fixes. Resolve host Codex auth/config from `process.env.CODEX_HOME || "~/.codex"` and only pin the container home separately.
47. **Ripgrep upstream only ships `.deb` packages for amd64.** `src/docker/assets/Dockerfile` must not derive a ripgrep `.deb` URL from `dpkg --print-architecture` (`arm64` 404s). Install `ripgrep` through Debian apt instead, or use the upstream tarball names explicitly per architecture. Regression coverage lives in `tests/docker.test.js`.
